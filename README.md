# swift-practice

Swift / SwiftUI を学習する過程で理解したことを、自分の言葉でまとめた学習用リポジトリ。
Apple 公式チュートリアル「Develop in Swift」などを進めながら書き溜めた個人的なメモです。

---

## Swift / SwiftUI の基本思想

- Swift は **"Clarity at the point of use"（使う場所での明快さ）** を重視する言語。コードを読んだときに、そこで何をしているかが伝わることを大事にする。
- **SwiftUI と Swift は別言語ではない。** 言語はどちらも Swift。違うのは「最初に渡されるテンプレート」だけ。
  - **SwiftUI** … 画面（見た目）を作るためのテンプレート／道具
  - **Swift（純粋な部分）** … データ・ロジック・モデルなどを記述する

## View と body の仕組み

- 処理は基本的に `main` から始まり、`body` を見にいく。`body` の中で処理（関数）を呼んだら、その定義を参照しにいく、というイメージ。
- `body` はあくまで **View（見た目）をまとめる場所**。
- `body` は**構造体（struct）の中に書く**という決まりがある。これは **宣言的UI** かつ **値型が前提**だから。Swift 言語の思想を SwiftUI の世界に徹底させた結果。
- View に関わる関数は2種類あるので注意：
  1. 表示のための計算処理（View 用の値を計算する）
  2. ユーザー操作用の関数（ボタンを押したときの処理など）
- 操作用の関数は「予約されているだけ」。呼ばれたら再び `body` に行き、再描画される。

## struct と関数

- C言語と違って、Swift は**構造体に関数（メソッド）を持たせられる**。
- 構造体を呼び出すときは、引数に各プロパティに対応する値を入れるだけでインスタンスが作れる。

## VStack とレイアウト

- `VStack` は、中に書いたビューを**きれいに縦に並べてくれるコンテナ**。コードとしても管理しやすく、間隔などの微調整も可能。
- **embed（埋め込み）** とは、既にあるビューを `VStack` などのコンテナで「囲んで、その中に入れる」こと。
  - Xcode では対象を **Control + クリック →「Embed in VStack」** で、既存のビューを丸ごと `VStack` の中に入れられる。
  - つまり「バラバラのビューを、1つの縦並びの箱にまとめる」操作。

## modifier（修飾子）

- **modifier の順番はとても重要。** 書いた順に上から適用され、行（順序）によって反映される場所・結果が変わる。
- 「テキストにパディング → その外に枠線」と「枠線 → その外にパディング」では結果が違う。順番に対してしっかりした仕組みがある。

### `.border` の使い方

- `.border` は**デザインとしても使える**が、**デバッグ用としても優秀**。
- 枠線があると、そのビューが占める作業領域を視覚化できる。レイアウトが思い通りにならないとき、どこがどれだけの領域を取っているか確認できる。

## 変数とプロパティ

- **変数（variable）** … 値を入れておく「入れ物」全般のこと。
- **プロパティ（property）** … 変数のうち、型（構造体・クラス・列挙型）の中に属しているもの。
- 使い分けの感覚：
  - 値・性質を表すなら → **プロパティ**
  - 動作を表すなら → **関数（メソッド）**
- 簡単な計算は、わざわざ関数を書いて呼び出すより、計算プロパティの中に書いたほうが自然に書ける（このあたりは慣れ）。

### 保存プロパティと計算プロパティ（※理解を修正した箇所）

- **保存プロパティ** … 値を実際に保持するプロパティ。`let`（変更しない）でも `var`（変更する）でもよい。
- **計算プロパティ** … 毎回その場で計算して値を返すプロパティ。必ず `var` で宣言する（getter を持つため `let` にはできない）。
- **ポイント**：`let` / `var` は「変わるか・変わらないか（可変性）」の区別であって、「保存か・計算か」の区別とは別軸。
  - 例：`@State private var keyword = ""` は「保存プロパティだが `var`」。変わる値だから `var`。

## @State と状態管理

- 普通の変数は、値が変化しても SwiftUI は検知できないし、再描画のたびに初期値に戻ってしまう。
- `@State` を付けると、その逆の性質になる：
  - 値の変化を SwiftUI が検知して再描画してくれる
  - 再描画をまたいでも値が保持される（初期値に戻らない）

### binding（未消化・あとで理解する）

- binding の概念はまだうまく理解できていないので、一旦スルーして進めている。
- 現時点の認識：入力系のビュー（`TextField` など）には binding を渡す。（`$` を付けて渡すやつ）

## プロトコル

- プロトコルを宣言（採用）すると、それに合わせた中身を定義しないといけない。
- 例：`Identifiable` というプロトコルを採用したら、`id` というプロパティを必ず持たなければならない。

## Assets（アセット）

- ダークモード対応などに使える、コードの外で**画像・色・素材などを管理する道具箱**、という認識。

## SwiftData

- SwiftData は **SQLAlchemy 的な立ち位置**（データをコードのモデルとして扱い、裏で DB に保存する仕組み）。
- SwiftData は中身で **SQLite** が動いている。

## struct と class の違い

- **struct** … プロパティを自動で初期化してくれる initializer が用意されている。変数は実体そのものを持つ（**値型**）。
- **class** … initializer を自分で `init()` 書く必要がある。変数はポインタ経由で heap 上の実体を指す（**参照型**）。
